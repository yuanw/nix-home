* Summary
  Serve [[https://github.com/AEON-7/Ornith-1.0-35B-AEON-Ultimate-Uncensored][Ornith-1.0-35B AEON Ultimate Uncensored]] on the DGX Spark
  (GB10 / sm_121a, aarch64-linux) using the official AEON vLLM container
  =ghcr.io/aeon-7/aeon-vllm-ultimate:latest= — the exact image the
  [[https://github.com/AEON-7/Ornith-1.0-35B-AEON-Ultimate-Uncensored/blob/main/QUICKSTART_DGX_SPARK.md][QUICKSTART_DGX_SPARK.md]] benchmarks against (NVFP4 + DFlash n=6 →
  ~93 tok/s avg decode, 3.05× vs BF16) — but driven *declaratively by
  NixOS* via systemd rather than ad-hoc =docker run=.

  This is the companion to the native (containerless) plan in
  [[file:ornith-dgx-spark-native.org][ornith-dgx-spark-native.org]]. The trade-off is the inverse of the native
  plan: **no multi-hour vLLM-from-source build and no AEON patches to
  maintain** (the container ships =qwen3_5_moe= + DFlash + sm_121a kernels
  pre-compiled — Gap A of the native plan *vanishes*), at the cost of
  introducing an OCI runtime + NVIDIA container toolkit / CDI on the
  Spark and accepting a non-reproducible third-party image as a runtime
  dependency.

  *Runtime choice — Podman + CDI, not Docker.* Although the AEON
  QuickStart and =serve_ornith.sh= write =docker run --gpus all=, the
  *validated GB10 NixOS path upstream* ([[https://github.com/graham33/nixos-dgx-spark/blob/main/modules/dgx-spark.nix][graham33/nixos-dgx-spark]]) uses
  **Podman with CDI** (=podman run --device nvidia.com/gpu=all=):
  =hardware.nvidia-container-toolkit.enable = true= alongside
  =virtualisation.podman.{enable,dockerCompat,dockerSocket}=. Our repo
  already leans Podman for dev ([[file:../modules/dev/podman.nix][modules/dev/podman.nix]]). We therefore
  translate the AEON recipe's =docker run --gpus all= to
  =podman run --device nvidia.com/gpu=all=. This keeps one runtime on the
  host, matches the proven GB10 CDI wiring, and lets the existing dev
  Podman module coexist. Docker remains an opt-out fallback only if an
  AEON-image quirk blocks Podman.

* Reuse assessment (from existing in-tree artifacts + the native plan)
  The container path reuses the *management* machinery built for the
  native plan and skips the *build* entirely. Nix lays down the systemd
  service and the model on disk; the container is just the executor.

  | Existing artifact | Reuse for the container path |

  *Verdict:* ~70% reuse on the management/wiring, ~30% new (Podman + NVIDIA
  CDI enablement + the thin container service module). The big win:
  **the vLLM build and AEON-patch maintenance work disappears entirely** —
  the single largest, longest-tail risk in the native plan (Gap A) is
  gone. The largest risk *here* is GB10 GPU passthrough into a container
  (Gap A), but graham33's block substantially de-risks it.

* Gaps to close (the only real work)

** Gap A — Podman + NVIDIA CDI on GB10 (port graham33's block)
   :PROPERTIES:
   :CUSTOM_ID: Gap A
   :END:
   The Spark currently is *not* an OCI-runtime host for inference:
   =virtualisation.docker.enable= is off (default in [[file:../modules/nixos_system.nix][modules/nixos_system.nix]] line 107),
   the [[file:../modules/dev/podman.nix][modules/dev/podman.nix]] module is a *dev* shell aid (not enabled
   system-wide on =dgx-spark=), and no NVIDIA container toolkit / CDI is
   wired despite the =docker= group being present on =yuanw=
   ([[file:../hosts/dgx-spark/configuration.nix][configuration.nix]] line 31). graham33's upstream =modules/dgx-spark.nix=
   (lines 164-174) is the *validated GB10 enablement* — port it almost
   verbatim:

   #+BEGIN_SRC nix :tangle no
   # Port of graham33/nixos-dgx-spark modules/dgx-spark.nix:164-174
   virtualisation.podman = {
     enable = true;
     dockerCompat = true;        # `docker` CLI alias → podman
     dockerSocket.enable = true; # docker-API socket for tools expecting dockerd
     defaultNetwork.settings.dns_enabled = true;
   };
   networking.firewall.trustedInterfaces = [ "podman+" ]; # reach host services
   hardware.nvidia-container-toolkit.enable = true;        # generates CDI spec
   #+END_SRC

   graham33 pairs this (same module, lines 138-153) with
   =hardware.nvidia = { modesetting.enable=true; open=true;
   nvidiaPersistenced=true; package = <production, scrubbed> }= — confirm
   our =dgx-spark= host already has the equivalent NVIDIA driver config
   (it must, since the native vLLM build already uses CUDA); if not, lift
   that block too.

   *CDI vs =--gpus all=:* with =hardware.nvidia-container-toolkit.enable=,
   NixOS generates =/etc/cdi/nvidia.yaml= (or the runtime variant). Use
   **=--device nvidia.com/gpu=all=** (the CDI form graham33's playbooks
   all use), *not* =--gpus all=. graham33's playbook shell hooks assert
   the CDI spec exists and print a fix hint
   (=sudo nvidia-ctk cdi generate --output=/etc/cdi/nvidia.yaml=) when
   missing — mirror those guards in the service =ExecStartPre=.

   *HF cache + image:* bind-mount =/var/lib/vllm/huggingface= (the module
   already creates it) as the container's HF home so any on-demand pulls
   inside the container share the host cache. Image pull can be an
   =ExecStartPre= =podman pull= (idempotent) or a separate oneshot; for
   reproducibility pin by digest
   =ghcr.io/aeon-7/aeon-vllm-ultimate@sha256:...=. The image is
   aarch64-targeted (matches GB10).

   *Forward-compat:* =serve_ornith.sh= already carries
   =NVIDIA_FORWARD_COMPAT=1= and =NVIDIA_DRIVER_CAPABILITIES=all= — keep
   both as =-e= env (see Gap B / Phase 3).

** Gap B — container service module
   :PROPERTIES:
   :CUSTOM_ID: Gap B
   :END:
   The existing [[file:../modules/vllm.nix][modules/vllm.nix]] hardcodes
   =ExecStart = "${pkgs.vllm-aeon or pkgs.vllm}/bin/vllm ...=.
   Two clean options:

   - *Option 1 (preferred, less churn):* add a =backend= option to the
     existing instance submodule (=native= | =podman=), and branch the
     =ExecStart= to either the binary or a =podman run= invocation. All
     the other options (=gpuMemoryUtilization=, =speculative=,
     =reasoningParser=, =extraEnv=, conflicts, drop-caches) carry over
     unchanged — DFlash, NVFP4, parsers, and env are identical between
     native and container. Only the entrypoint command differs.
   - *Option 2:* a sibling =modules/vllm-container.nix= mirroring the
     option set. More duplication; only justified if the two backends
     diverge more than expected.

   The =podman run= =ExecStart= shape (AEON QuickStart translated to CDI):

   #+BEGIN_SRC bash :tangle no
   podman run --rm --name vllm-ornith \
     --device nvidia.com/gpu=all --ipc host --net host \
     -e TORCH_CUDA_ARCH_LIST=12.1a \
     -e ENABLE_NVFP4_SM100=0 \
     -e VLLM_USE_FLASHINFER_SAMPLER=1 \
     -e NVIDIA_FORWARD_COMPAT=1 \
     -e NVIDIA_DRIVER_CAPABILITIES=all \
     -v /var/lib/vllm/models/Ornith-1.0-35B-NVFP4:/model:ro \
     -v /var/lib/vllm/models/AEON-DFlash-Qwen3.6-35B-A3B:/drafter:ro \
     --entrypoint vllm \
     ghcr.io/aeon-7/aeon-vllm-ultimate:latest \
     serve /model --served-model-name ornith \
       --quantization compressed-tensors \
       --speculative-config '{"method":"dflash","model":"/drafter","num_speculative_tokens":6}' \
       --gpu-memory-utilization 0.6 --max-model-len 262144 \
       --max-num-seqs 16 --max-num-batched-tokens 16384 \
       --mamba-cache-dtype float32 --reasoning-parser qwen3 \
       --enable-auto-tool-choice --tool-call-parser qwen3_coder \
       --enable-prefix-caching --trust-remote-code \
       --host 0.0.0.0 --port 8003
   #+END_SRC

   Container-module specifics vs the native branch:
   - =podman run= (not =docker run=); =--device nvidia.com/gpu=all= (CDI,
     not =--gpus all=), =--ipc host --net host= — copied verbatim from
     graham33's validated GB10 playbooks.
   - Bind-mount model dirs rather than referencing host paths in args.
   - Pass GB10 env via =-e= (the native branch sets process =Environment=).
   - =--network host= means =--port= inside the container binds the host
     port directly — no =-p= mapping needed (matches the QUICKSTART).
   - Use =--rm= with the systemd unit owning lifecycle (=Restart=on-failure=);
     do *not* =-d= (systemd is the foreground supervisor).
   - =ExecStartPre=: guard that the CDI spec exists (mirror graham33's
     shellHook check) and =podman pull ${image}= (idempotent, cheap when
     cached).
   - Reuse the existing model oneshot instances; the vLLM unit =requires=
     =vllm-model-Ornith-1.0-35B-NVFP4.service= etc. so files exist before
     =podman run=.

** Gap C — KV cache + Drafter choice (same as native, restated)
   - **BF16 KV** (vision tower forces non-causal attn). Do NOT pass
     =--kv-cache-dtype fp8=. Either omit it or pass =--kv-cache-dtype= off.
   - **AEON all-full-attention drafter** =AEON-7/AEON-DFlash-Qwen3.6-35B-A3B=
     so the SWA KV-page assert (issue #1) never fires and the
     =hotfixes/ornith_dflash_kvfix/= patch is *not needed*. Higher
     acceptance than the z-lab drafter (3.71 vs 3.35 tok/step).

* Phase 2 — Thin container backend in the instance module
  1. Add =backend= option to [[file:../modules/vllm.nix][modules/vllm.nix]] instance submodule (default =native=):
     #+BEGIN_SRC nix :tangle no
     backend = mkOption {
       type = types.enum [ "native" "podman" ];
       default = "native";
       description = "Run the native vllm-aeon binary or the AEON OCI image via podman.";
     };
     containerImage = mkOption {
       type = types.nullOr types.str;
       default = "ghcr.io/aeon-7/aeon-vllm-ultimate:latest";
       description = "Image used when backend=podman.";
     };
     #+END_SRC
  2. In =mkService=, branch =ExecStart=:
     - =native= → unchanged (=${pkgs.vllm-aeon or pkgs.vllm}/bin/vllm ${args}=).
     - =podman= → =${pkgs.podman}/bin/podman run --rm --name vllm-${name}
       --device nvidia.com/gpu=all --ipc host --net host= + =-e= for each
       =extraEnv= entry + =-v ${inst.model}:/model:ro= (and =/drafter= for
       the speculative model) + =--entrypoint vllm ${inst.containerImage}
       serve /model ...=, with the host model path stripped from args
       (replaced by =/model=).
  3. Reuse everything else verbatim: =conflicts= (single GPU), =drop_caches=
     =ExecStartPre=, =Restart=on-failure=, =requires= model-oneshot units.
  4. Container =ExecStartPre=: =podman pull ${inst.containerImage}= (idempotent,
     cheap when cached) and a CDI-spec existence guard (mirror graham33).
  5. Keep the =mambaCacheDtype= / GB10-env-default additions from the
     native plan's Gap B — they apply to both backends (the container
     ignores host env when =-e= is supplied, so set them via =-e=).

* Operating notes (from QUICKSTART + container specifics)
  - DFlash acceptance decays over many hours; =systemctl restart
    vllm-ornith= periodically (or a timer) restores it.
  - For higher batch throughput: run **plain NVFP4, DFlash off** and
    raise =max-num-seqs= — never raise it above ~16 with DFlash on.
  - Image drift: =latest= is convenient but non-reproducible. Pin by
    digest (=ghcr.io/aeon-7/aeon-vllm-ultimate@sha256:...=) once a known-
    good build is validated; bump deliberately.
  - Disk: keep ~30 GB free for the image + Drafter; the NVFP4 model lives
    under =/var/lib/vllm/models= (bind-mounted, not copied into the
    container layer) so it is shared with the native path.
  - Podman vs Docker: stay on Podman+CDI to match graham33's validated GB10
    wiring and the existing [[file:../modules/dev/podman.nix][modules/dev/podman.nix]] lean. Only flip to
    Docker if an AEON-image quirk blocks Podman (note it in R1).
  - Uncensored model (0/80 refusals): keep port 8003 off public ingress;
    use existing wireguard/age-secret-managed HF token for downloads.

* Native vs Container — quick decision matrix
  | Aspect | Native ([[file:ornith-dgx-spark-native.org][native plan]]) | Container (this plan) |
  |---------------------------------------+-----------------------------+------------------------------------|
  | Largest risk / work | vLLM-from-source build + AEON patch maintenance (Gap A) | GB10 GPU passthrough / CDI (Gap A) — largely de-risked by graham33's block |
  | Reproducibility | High (pinned nixpkgs + patches) | Medium (image =latest= by tag; pin digest to fix) |
  | First-time time-to-serve | Multi-hour vLLM build | ~image pull time (~minutes once Podman + CDI work) |
  | Patch upkeep | Rebase AEON patches on each vLLM bump | Pull new AEON image; zero local patch work |
  | Binary-cache dependency | Cachix (graham33 / own) | ghcr.io (AEON) |
  | OCI runtime | none | Podman + CDI (graham33-validated); Docker as fallback |
  | Service shape | systemd → native binary | systemd → =podman run= |
  | Model on disk (=vllm-models= module) | Reused | Reused (bind-mounted) |
  | Validated perf envelope (NVFP4 + DFlash n=6, 0.6, seqs≤16) | Identical | Identical (same flags, same image QuickStart used) |
