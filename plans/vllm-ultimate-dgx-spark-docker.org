* Summary
  Serve the [[https://github.com/AEON-7/vllm-ultimate-dgx-spark#quickstart-dgx-spark-copy-paste][vLLM Ultimate DGX Spark]] fleet — **Qwen3.6-27B**, **Gemma-4-26B-A4B**, and
  **Qwen3.6-35B-A3B** — on the DGX Spark (GB10 / sm_121a, aarch64-linux) using the
  *same* [[https://ghcr.io/aeon-7/aeon-vllm-ultimate][AEON vLLM container]] =ghcr.io/aeon-7/aeon-vllm-ultimate:latest= validated
  in the ornith plan, but driven *declaratively by NixOS via systemd + Podman*
  rather than ad-hoc =docker run=.

  This is the **fleet companion** to the
  [[file:ornith-dgx-spark-docker.org][ornith-dgx-spark-docker.org]] plan (which deploys the Ornith-1.0-35B AEON model only).
  Whereas ornith is a single-model monolith (NVFP4 + AEON DFlash drafter, BF16 KV,
  port 8003), this plan deploys the *three-model fleet* benchmarked in the
  upstream README — each with its own port and optimal DGX Spark configuration:

  | Instance | Model | Drafter | KV | Port | Spec tokens | GPU util |
  |----------+-------+---------+----+------+-------------+----------|
  | =aeon= | Qwen3.6-27B NVFP4 | z-lab DFlash (5-layer) | FP8 | 8000 | 12 | 0.78 |
  | =gemma= | Gemma-4-26B-A4B NVFP4 | Gemma DFlash (5-layer, SWA) | FP8 | 8001 | 12 | 0.78 |
  | =qwen35b= | Qwen3.6-35B-A3B NVFP4 | Qwen3.6-35B DFlash (8-layer) | FP8 | 8002 | 12 | 0.76 |

  *Reuse note:* the three =native= instances already exist in
  [[file:../hosts/dgx-spark/configuration.nix][hosts/dgx-spark/configuration.nix]] (→ [[file:../hosts/dgx-spark/configuration.nix][configuration.nix]]). The Podman+CDI
  infrastructure is proven by the =ornith= container instance (verified 2026-06-28).
  The only work needed per fleet model is: add =backend = "podman"=, bind the
  correct container image, confirm the KV/dtype flags match the QuickStart, and
  re-validate throughput.

** Gap B — Add =--shm-size=16g= to the podman backend args
   :PROPERTIES:
   :CUSTOM_ID: Gap B
   :END:

   The upstream QuickStart's =docker run= includes =--shm-size=16g=
   (shared memory for vLLM's CUDA IPC / PyTorch distributed). The
   current =podman= backend in [[file:../modules/vllm.nix][modules/vllm.nix]] does not set this.
   Podman defaults to 64 MB =/dev/shm= — too small for vLLM's CUDA
   graphs and tensor-parallel IPC (even though these models are
   single-GPU, vLLM still uses shared memory for the async scheduler
   and CUDA events).

   Add to the =podmanArgs= list in =mkService=:

   #+BEGIN_SRC nix :tangle no
   ++ [ "--shm-size" "16g" ]
   #+END_SRC

   Placement: after =--network host=, before =-e= flags. Podman's CLI
   accepts both =--shm-size 16g= and =--shm-size=16g= — the Nix
   =escapeShellArgs= serializes them identically either way. Stick
   with the two-arg form for readability (matching how =--device= and
   =--ipc= are already written).

   *Why this was missed in the ornith plan:* the ornith instance's
   QUICKSTART_DGX_SPARK.md also uses =--shm-size= = (etymology: the
   =serve_ornith.sh= recipe). We simply need to add it — both the
   ornith plan and this fleet plan benefit.

   *Verification:* after adding =--shm-size 16g=, check in the running
   container:
   #+BEGIN_SRC bash :tangle no
   podman exec vllm-aeon df -h /dev/shm   # should show 16G
   #+END_SRC

   Or, if the container does not expose a shell exec-able entrypoint,
   check the container inspect output:
   #+BEGIN_SRC bash :tangle no
   podman inspect vllm-aeon --format '{{.HostConfig.ShmSize}}'
   # → 17179869184 (16 GiB in bytes)
   #+END_SRC

* Phase 3 — Deploy + smoke test

  1. =nixos-rebuild switch --flake .#dgx-spark=
  2. If model downloads haven't run yet (fresh host):
     #+BEGIN_SRC bash :tangle no
     systemctl start vllm-model-Qwen3.6-27B-AEON-NVFP4
     systemctl start vllm-model-Qwen3.6-27B-DFlash-drafter
     #+END_SRC
     Wait for both to =RemainAfterExit=.
  3. Start the instance:
     #+BEGIN_SRC bash :tangle no
     systemctl start vllm-aeon
     journalctl -u vllm-aeon -f
     #+END_SRC
     Expected boot: vLLM v0.23.0 loading the AEON image, DFlash
     drafter initializing, NVFP4 weights, then /Application startup
     complete/.
  4. Smoke test (upstream QuickStart §5):
     #+BEGIN_SRC bash :tangle no
     curl -s http://dgx-spark.local:8000/v1/chat/completions \
       -H "content-type: application/json" \
       -d '{"model":"aeon","messages":[{"role":"user","content":"Hello!"}],"max_tokens":64,"temperature":0.0}' \
       | python3 -m json.tool
     #+END_SRC
     Expect coherent response; ~30-50 tok/s single-stream.
  5. Repeat for =gemma= (port 8001) and =qwen35b= (port 8002).

* Operating notes

  - **One instance at a time:** instances declare mutual =conflicts=;
    starting =vllm-aeon= stops =vllm-gemma= and vice-versa (single GPU).
    Use =conflicts= override only if deliberately running CPU-offloaded
    instances (not supported).
  - **DFlash acceptance decays** over many hours of sustained traffic;
    =systemctl restart vllm-<name>= restores it. Consider a daily timer
    if the instance runs continuously.
  - **Container image drift:** =:latest= is convenient but
    non-reproducible. Pin by digest
    (=ghcr.io/aeon-7/aeon-vllm-ultimate@sha256:...=) once a known-good
    build is validated; bump deliberately (the ornith plan verified a
    successful pull 2026-06-28 — use that digest).
  - **=--shm-size=16g=** is now wired in the podman backend — no
    per-instance knob needed. Podman mounts the shm regardless of
    =--device nvidia.com/gpu=all=.
  - **Model downloads** are shared between native and container paths
    (same =/var/lib/vllm/models=). Switching backend does not re-download.
  - **KV cache dtype:** the fleet uses =fp8_e4m3= (FP8), not =nvfp4=.
    The upstream README explains that DFlash (non-causal) does not pair
    with NVFP4 KV on GB10 today. If using MTP self-speculation instead
    of DFlash, switch to =nvfp4= for ~3× KV capacity (Recipe B).
  - **=--max-num-seqs 8=** (two instances) or =16= (qwen35b) — well below
    the DFlash crash threshold of 32 on GB10 (the v0.23.0 image carries
    the high-concurrency fix that scales to 64, so these are safe).
  - **Unified memory:** =gpu-memory-utilization= never exceeds 0.78
    (qwen35b is 0.76). On GB10, the LPDDR5X pool is shared between CPU
    and GPU — exceeding ~0.88 page-thrashes.

* Native vs Container — fleet decision matrix

  | Aspect | Native (current state) | Container (this plan) |
  |---------------------------------------+---------------------------+--------------------------------|
  | vLLM binary | =pkgs.vllm-aeon= (built from source, multi-hour) | AEON container (pre-compiled sm_121a + DFlash) |
  | Maintenance | Rebase AEON patches on vLLM bumps | Pull new AEON image; zero local patch work |
  | Build time | Multi-hour (GB10 aarch64) | None (pre-built image) |
  | GPU passthrough | Native (no OCI runtime) | Podman + CDI (proven) |
  | Reproducibility | High (pinned nixpkgs) | Medium (=latest= tag); pin digest to fix |
  | Shm size | Default (systemd) | =--shm-size=16g= (explicit, matching QuickStart) |
  | Deployment overhead | None beyond NixOS | Image pull + CDI verification |
  | DFlash + NVFP4 KV | Not yet possible (vLLM builds may diverge) | Container ships working combination |
  | Fleet validation | Individual model testing | Single image validated across all three models |

* Risks / open questions

  | # | Question | Mitigation |
  |---+----------+------------|
  | R1 | =--shm-size=16g= regression — does vLLM actually need 16 GB on single-GPU aarch64, or is 2 GB enough? | Start with 16 GB (matching QuickStart); reduce to 2 GB if systemd =MemoryMax= pressure is observed. The QuickStart value is conservative for =--shm-size=. |
  | R2 | Image =:latest= may drift between ornith validation (2026-06-28) and fleet deployment | Pin to the same digest the ornith plan pulled (recorded in ornith-dgx-spark-docker.org Phase 1 verification table) |
  | R3 | Gemma-4-26B has 5 layers of SWA in the DFlash drafter (4 of 5 SWA-2048) — does the FP8 KV change anything vs the ornith instance's BF16 KV for the Gemma drafter? | No — =--kv-cache-dtype= controls the *base* model's KV, not the drafter's. The Gemma drafter's SWA operates independently of the KV cache dtype. Confirmed by upstream README fleet benchmarks (all three use FP8 KV). |
  | R4 | =--shm-size= flag in the native backend: native vLLM is already running without it; is it only a container concern? | Yes —=--shm-size= is an OCI-runtime-specific flag (=/dev/shm= mount size). Native systemd services use the host's =/dev/shm= (typically 50% of RAM). Only affects the podman backend. |
  | R5 | Podman backend sets =--network host= — does this bypass the firewall rules (allowedTCPPorts 8000-8002)? | =--network host= means the container uses the host's network stack directly, so host firewall rules apply. The =networking.firewall.allowedTCPPorts= rules already cover all three ports. Verified by ornith (port 8003 works). |
