** Key Decisions

  - *Model download at runtime*: Models are large (~20GB+) and not in Nix store;
    =lance-download-model= (standalone, huggingface-hub only) runs on target machine.
  - *Python env via =python3.withPackages=*: Self-contained closure.
  - *Multi-instance architecture*: =lance-gradio-video= (port 7860) + =lance-gradio-image=
    (port 7861) with independent model/task/port settings.
  - *decord lazy import*: Since decord is hard to build from source (FFmpeg API
    incompat), made import lazy. Gradio server starts without decord; video
    tasks that need it will fail until decord is available.
  - *flash-attn*: Required for all Lance inference. Build blocked by g++ 15 / CUDA 12.9
    incompatibility. =backendStdenv= should handle this but doesn't propagate correctly.

** 2.2.  Nix Packaging Strategy
   For a Nix flake, the Lance model can be packaged as a derivation that
   downloads weights at build time and produces a FHS-compatible environment.

*** 2.2.1.  =lance-model-weights= Derivation
    #+begin_src nix
    {
      lance-model-weights = stdenv.mkDerivation {
        name = "lance-model-weights";
        src = fetchurl {
          # URLs for HuggingFace safetensors
          # Use HF mirror or direct URLs
        };
        installPhase = ''
          mkdir -p $out/Lance_3B $out/Lance_3B_Video
          # copy/move weights into $out structure
        '';
      };
    }
    #+end_src

    /Alternative:/ Use =fetchFromHuggingFace= from =huggingface.nix= or
    =fetchHuggingFaceModel= if available.

*** 2.2.2.  =lance-python-env= Derivation
    A Python environment with all dependencies, using =poetry2nix= or
    =mach-nix= or =python-with-packages= with overrides for ARM64.

    Key overrides needed:
    - =torch=:  use =torch-bin= for ARM64 CUDA support
    - =bitsandbytes=:  use =bitsandbytes-bin= or build from source
    - =flash-attn=:  build from source with ARM64 patches
    - =triton=:  use =triton-bin= for ARM64

*** 2.2.3.  =lance-app= Combined Derivation
    Combines model weights + Python env + entry script into a single
    package that can be run as =lance-app/run.sh=.

    Entry point script (simplified):
    #+begin_src bash :tangle lance-app/run.sh
    #!/bin/sh
    cd "$LANCE_ROOT"
    exec accelerate launch \
      --num_machines 1 \
      --num_processes 1 \
      --mixed_precision bf16 \
      inference_lance.py \
      --model_path "$MODEL_PATH" \
      --vit_type qwen_2_5_vl_original \
      ...  # rest of args
    #+end_src

** 2.3.  DGX-Spark-Specific Packaging

*** 2.3.1.  Container (OCI) Image
    For DGX Spark, Docker / Podman container is the standard deployment
    unit.  Use =nix2container= or =dockerTools= to produce a container
    image.

    Container layers:
    1. Base: =nvidia/cuda:12.4-runtime-ubuntu24.04-arm64= or DGX-provided
       base
    2. Python env (from =lance-python-env=)
    3. Model weights (from =lance-model-weights=)
    4. Entry script + Gradio UI

    #+begin_src dockerfile :tangle Dockerfile
    FROM lance-python-env:latest AS base
    COPY --from=lance-model-weights /weights /lance/weights
    COPY lance-app/ /lance/app/
    WORKDIR /lance/app
    EXPOSE 7860
    CMD ["python3", "lance_gradio_t2v_v2t.py", "--gpus", "0"]
    #+end_src

*** 2.3.2.  FHS User Env (Non-Container)
    For direct NixOS / nix-darwin deployment without containers, create a
    FHS-compatible =buildFHSUserEnv= that provides:
    - CUDA libraries from =cudaPackages=
    - Python + deps from =lance-python-env=
    - Model weights symlinked from =lance-model-weights=
    - =lance.sh= entry script

*** 2.3.3.  Model Variant Selection
    The user should be able to select between =Lance_3B= (image tasks) and
    =Lance_3B_Video= (video tasks).  The packaging should expose both as
    configurable =model_path= arguments.

    Recommended: package both variants, let the service configuration choose.

* 3.  Systemd Service Design

** 3.1.  Service Architecture
   Two possible service modes:

*** 3.1.1.  Gradio Server (Interactive)
    Uses [[file:/tmp/Lance/lance_gradio_t2v_v2t.py][lance_gradio_t2v_v2t.py]] – provides a web UI on port 7860.

    Characteristics:
    - Gradio Blocks interface with text/video/image inputs
    - Loads model once, keeps in GPU memory
    - Queue-based request handling (configurable queue size)
    - Supports t2v, v2t (x2t_video) tasks
    - Can be extended for all 6 task types

*** 3.1.2.  API Server (Programmatic)
    A lightweight REST/gRPC server wrapping =inference_lance.py= for
    programmatic access.  Not provided upstream; must be implemented.

    Proposed API endpoints:
    - =POST /api/v1/generate= – generation tasks
    - =POST /api/v1/understand= – understanding tasks
    - =GET /api/v1/health= – health check
    - =POST /api/v1/config= – runtime parameter adjustment

** 3.2.  Systemd Unit File

*** 3.2.1.  Gradio Server Unit
    #+begin_src ini :tangle lance-gradio.service
    [Unit]
    Description=Lance Multimodal AI Model – Gradio Server
    After=network-online.target
    Wants=network-online.target

    [Service]
    Type=simple
    User=lance
    Group=lance
    ExecStart=${LANCE_ROOT}/lance-gradio.sh
    Restart=on-failure
    RestartSec=10
    StandardOutput=journal
    StandardError=journal
    Environment=PYTHONWARNINGS=ignore
    Environment=CUDA_LAUNCH_BLOCKING=0
    Environment=NCCL_DEBUG=VERSION
    Environment=POSITION_EMBEDDING_3D_VERSION=v2

    # GPU isolation – bind to first GPU
    PrivateDevices=true
    # Adjust if using container runtime:
    # Runtime using cgroup device access for GPU
    SupplementaryGroups=cuda

    [Install]
    WantedBy=multi-user.target
    #+end_src

*** 3.2.2.  Environment File (=/etc/lance/lance.conf=)
    #+begin_src ini :tangle lance.conf
    # Lance Model Configuration
    MODEL_PATH=/nix/store/...lance-3b-video/Lance_3B_Video

    TASK_NAME=t2v
    NUM_GPUS=1
    NUM_FRAMES=50
    VIDEO_HEIGHT=480
    VIDEO_WIDTH=848
    RESOLUTION=video_480p
    VALIDATION_NUM_TIMESTEPS=30
    VALIDATION_TIMESTEP_SHIFT=3.5
    CFG_TEXT_SCALE=4.0
    USE_KVCACHE=true
    TEXT_TEMPLATE=true
    SERVER_PORT=7860
    GRADIO_TASK=t2v
    QUEUE_SIZE=32
    #+end_src

*** 3.2.3.  Entry Script (=lance-gradio.sh=)
    #+begin_src bash :tangle lance-gradio.sh
    #!/bin/sh
    set -e
    cd "${LANCE_ROOT:-/lance/app}"
    exec python3 lance_gradio_t2v_v2t.py \
      --gpus "${GPU_DEVICE:-0}" \
      --server-port "${SERVER_PORT:-7860}" \
      --model-path "${MODEL_PATH}" \
      --vit-type "${VIT_TYPE}" \
      --task "${GRADIO_TASK}" \
      --timesteps "${VALIDATION_NUM_TIMESTEPS}" \
      --timestep-shift "${VALIDATION_TIMESTEP_SHIFT}" \
      --cfg-text-scale "${CFG_TEXT_SCALE}" \
      --resolution "${RESOLUTION}" \
      --height "${VIDEO_HEIGHT}" \
      --width "${VIDEO_WIDTH}" \
      --num-frames "${NUM_FRAMES}" \
      --queue-size "${QUEUE_SIZE}"
    #+end_src

** 3.3.  Model Loading & Warm-Up
   The Gradio server pre-loads the model at startup.  To avoid timeout,

   #+begin_src systemd

   #+end_src

** 3.4.  Multi-GPU / Multi-Instance
   DGX Spark has a single powerful GPU.  For serving multiple task types
   concurrently, consider:

   - Instance 1: Gradio server for t2v/v2t (primary)
   - Instance 2: API server for t2i/image_edit (secondary)
   - GPU memory sharing via MIG if Blackwell supports it, or sequential
     task queuing

   For multi-instance, create separate systemd services:
   - =lance-gradio-t2v.service=
   - =lance-api-image.service=

** 3.5.  GPU Isolation & Resource Limits
   #+begin_src systemd
   [Service]
   # Prevent OOM kills on model reload
   OOMScoreAdjust=-500
   # GPU device access
   DeviceAllow=/dev/nvidia*
   # Memory limit (adjust based on VRAM)
   MemoryMax=96G
   #+end_src

* 6.  Operational Runbook

** 6.1.  First-Time Setup
   1. Build the flake: =nix build .#dgx-spark=
   2. Copy model weights (or fetch via =nix build .#lance-model-weights=)
   3. Enable systemd service: =systemctl enable lance-gradio=
   4. Start service: =systemctl start lance-gradio=
   5. Check logs: =journalctl -u lance-gradio -f=
   6. Open UI at =http://<dgx-spark-ip>:7860=

** 6.2.  Model Update
   1. Update the HuggingFace model reference in flake
   2. Rebuild: =nix build .#dgx-spark=
   3. Restart: =systemctl restart lance-gradio=
   4. Verify new model loads correctly

** 6.3.  Task Switching
   To switch between image tasks and video tasks:
   - Change =MODEL_PATH= in =/etc/lance/lance.conf=
   - Restart service: =systemctl restart lance-gradio=
   - Or use the API endpoint =POST /api/v1/config= to switch at runtime
     (if implemented)

** 6.4.  Troubleshooting
   | Symptom                              | Likely Cause                 | Fix                                                  |
   |--------------------------------------+------------------------------+------------------------------------------------------|
   | OOM on startup                      | Insufficient VRAM            | Lower =max_latent_size=, reduce resolution           |
   | =CUDA error=                       | CUDA version mismatch        | Verify torch+CUDA compatibility on ARM64             |
   | =FileNotFoundError: model.safetensors=| Missing model weights        | Run =nix build .#lance-model-weights=                |
   | Gradio UI blank                     | Port conflict                | Change =SERVER_PORT=                                 |
   | Slow inference                     | No flash-attn on ARM64       | Build flash-attn for ARM64 or use xformers           |

* Appendix: Lance Architecture Notes

** Model Components
   Lance ([[file:/tmp/Lance/modeling/lance/lance.py]]) is built from:

   | Component                     | Source                                                       |
   |-------------------------------+-------------------------------------------------------------|
   | LLM Backbone (Qwen2ForCausalLM)| [[file:/tmp/Lance/modeling/lance/qwen2_navit.py]] – Qwen2    |
   |                               | with MoE + NaViT sparse attention                            |
   | Vision Encoder (ViT)          | [[file:/tmp/Lance/modeling/vit/qwen2_5_vl_vit.py]] –         |
   |                               | Qwen2.5-VL Vision Transformer                                |
   | VAE Decoder                   | [[file:/tmp/Lance/modeling/vae/wan/model.py]] – Wan VAE      |
   | Connector                     | [[file:/tmp/Lance/modeling/lance/modeling_utils.py]] –       |
   |                               | MLPconnector                                                 |
   | Position Embedding            | [[file:/tmp/Lance/modeling/lance/modeling_utils.py]] –       |
   |                               | 3D sinusoidal position embeddings                            |
   | Timestep Embedder             | [[file:/tmp/Lance/modeling/lance/modeling_utils.py]] –       |
   |                               | sinusoidal timestep embeddings                               |

** Inference Flow
   Generation (t2v, t2i, image_edit, video_edit):
   1. Tokenize prompt, create interleave sequence (text + VAE/ViT tokens)
   2. Encode condition images via ViT (if editing task)
   3. Encode condition videos via VAE (if editing task)
   4. Run flow-matching denoising loop (=num_timesteps= steps)
   5. CFG-guided sampling (text_scale, vit_scale)
   6. Decode VAE latent back to pixels → save as PNG/MP4

   Understanding (x2t_image, x2t_video):
   1. Tokenize prompt with ViT image/video tokens
   2. Autoregressive text generation (=max_length= tokens)
   3. Decode generated token IDs → answer text

** Key Parameters
   | Parameter                 | Default | Range           | Effect                              |
   |---------------------------+---------+-----------------+-------------------------------------|
   | =num_timesteps=           | 30      | 10–100          | Quality vs. speed                   |
   | =timestep_shift=          | 3.5     | 1.0–10.0        | Noise schedule shift                |
   | =cfg_text_scale=          | 4.0     | 1.0–10.0        | Prompt adherence vs. creativity     |
   | =num_frames=              | 50      | 1–121           | Video length (max 121 @ ~480p)      |
   | =resolution=              | 480p    | 192p, 360p, 480p| Spatial resolution                  |
   | =video_height/width=      | 768     | variable        | Output dimensions                   |
   | =use_KVcache=             | true    | true/false      | KV cache for understanding speed    |
